# AI風に見えないnote記事の作り方

## 基本方針

note記事は、AIが完成させるのではなく、社長の体験談・失敗談・検証結果・感情の動きをAIが編集する形で作る。

体験していないことを体験談風に創作しない。  
実体験が足りない場合は、検証ログ、仮説記事、または架空ケーススタディとして明示して書く。

AIの役割:

- 読者の悩みを整理する。
- 記事構成を作る。
- 社長に聞くべき体験素材を質問にする。
- 入力された体験談を読みやすく整える。
- 事実、意見、体験談、引用を分ける。

社長の役割:

- 実際に体験したことを入れる。
- 失敗したこと、迷ったこと、判断したことを入れる。
- 数字、時系列、具体例を入れる。
- 最後に自分の言葉でリライトする。

## 実体験が足りない場合の代替

| 方法 | 使い方 | 表記ルール |
| --- | --- | --- |
| 検証ログ | 実際に短期間で試したことを書く | 「実際に試した範囲では」と明記する |
| 仮説記事 | 調査から見えた仮説を書く | 「現時点の仮説」と明記する |
| 架空ケーススタディ | 読者像を想定して手順を説明する | 「架空ケース」「想定例」と明記する |
| 匿名化した読者ケース | 実在の相談を個人が特定できない形にする | 個人情報を除き、本人が分からないようにする |

NG:

```text
私はこの方法で月10万円稼ぎました。
```

OK:

```text
実際に3日間、noteランキングを見てネタ候補を20個出してみたところ、反応が取りやすそうな型は3つありました。
```

OK:

```text
架空ケースとして、ブログで挫折した会社員Aさんを想定すると、最初にやるべきことは自己紹介記事ではなく、代表作記事の設計です。
```

## AI風に見える記事の特徴

- どこかで見たような一般論が続く。
- 主語が大きい。
- 具体的な人物、場面、日時、数字がない。
- 成功談だけで失敗や迷いがない。
- 感情の動きがない。
- 読者への次アクションが抽象的。
- 見出しが「〇〇とは」「メリット3選」「まとめ」だけで終わる。

## 体験談として入れる素材

| 素材 | 例 |
| --- | --- |
| きっかけ | なぜそのテーマに取り組んだのか |
| 失敗 | 何を間違えたのか |
| 迷い | どこで判断に悩んだのか |
| 試行錯誤 | 何を試して、何をやめたのか |
| 数字 | 期間、回数、金額、ビュー数、作業時間 |
| 会話 | 誰に何を言われたのか |
| 感情 | 焦り、不安、納得、違和感、手応え |
| 気づき | やってみて初めて分かったこと |
| 現在地 | 今はどう運用しているのか |

## 制作フロー

### Step 1: AIが読者の悩みを1人に絞る

悪い例:

```text
副業を始めたい人向け
```

よい例:

```text
ブログを1年続けたが月数百円しか稼げず、WordPressの保守にも疲れて、noteに移るべきか迷っている30代会社員
```

### Step 2: AIが社長に体験質問を出す

記事を書く前に、AIは社長へ以下を確認する。

```markdown
## 体験素材ヒアリング

- このテーマで実際にやったことは何ですか？
- 最初に失敗したことは何ですか？
- どの時点で考え方が変わりましたか？
- 数字で出せる変化はありますか？
- 読者に同じ失敗をさせないために、何を伝えたいですか？
- 今なら最初に何をしますか？
```

### Step 3: AIが構成を作る

推奨構成:

```markdown
# タイトル

## この記事は誰向けか

## 先に結論

## 私がこのテーマでつまずいたこと

## 実際にやったこと

## うまくいかなかったこと

## 変えたこと

## 変えてから起きた変化

## ここは注意した方がいい

## 読者が次にやること
```

### Step 4: AIが本文化する

AIは社長の体験素材を中心に本文化する。  
体験素材が足りない箇所は、無理に創作せず `[社長が追記]` と書く。

### Step 5: 社長が最終リライトする

社長は以下を直す。

- 言い回しが自分らしいか。
- 体験談が薄い箇所はないか。
- AIっぽい一般論が残っていないか。
- 読者への次アクションが具体的か。
- 誇大表現や断定がないか。

## AIに禁止すること

- 体験していないことを体験談風に書く。
- 数字を作る。
- 読者の悩みを広げすぎる。
- 失敗談を美談にしすぎる。
- 「多くの人が」「誰でも」「必ず」などの大きすぎる主語で断定する。
- 競合記事の構成をそのまま再利用する。
- 架空ケースを実在の体験のように見せる。

## 記事完成前のチェック

| チェック項目 | 判定 |
| --- | --- |
| 読者が1人に絞られているか | OK / NG |
| 社長の体験談が入っているか | OK / NG |
| 失敗・迷い・試行錯誤が入っているか | OK / NG |
| 数字・時系列・具体例があるか | OK / NG |
| AIが作った架空体験がないか | OK / NG |
| 架空ケースは架空と明示されているか | OK / NG |
| 仮説は仮説として書かれているか | OK / NG |
| 検証範囲が明記されているか | OK / NG |
| 一般論だけの段落が残っていないか | OK / NG |
| 最後の次アクションが具体的か | OK / NG |

1つでもNGがある場合、公開しない。
