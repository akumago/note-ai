# AI会社組織システム 要件定義

## 1. システム概要

| 項目 | 内容 |
| --- | --- |
| 目的 | 副業コンテンツ制作・販売ワークフローの自動化 |
| 実行環境 | Codex / Claude Code（CLIエージェント） |
| 操作インターフェース | Codex Web UI、Codexモバイル、LINE Bot（社長秘書部署） |
| 前提契約 | ChatGPT Pro（月$20）またはClaude Max |

## 2. 組織階層

```text
社長（人間）
  └── 部長（AIエージェント、部署ごとに1体）
        └── 課長（AIエージェント、機能ごとに1体）
              └── 実行部隊（AIエージェント、複数体）
```

### 階層ルール

- 各層は上位からのみ命令を受ける。
- 各層は成果物を上位に報告し、決済を仰ぐ。
- 社長（人間）が触るのは原則として「採用 / 却下」の決済のみとする。

## 3. 部署別機能要件

### 部署1: リサーチ部

| 要件 | 内容 |
| --- | --- |
| 構成 | 部長1・アナリスト3 |
| トリガー | 毎朝7時（Automation機能） |
| インプット | 競合チャンネル名、調査対象キーワード |
| アウトプット | 競合YouTube新着まとめ / Xトレンドまとめ / Note売れ筋上位10本まとめ（各レポートMarkdown形式） |
| 決済 | 社長が採用 / 却下を返す |

### 部署2: 制作部

| 要件 | 内容 |
| --- | --- |
| 構成 | 編集長1・ライター複数 |
| トリガー | リサーチ部レポート決済後（朝9時） |
| インプット | リサーチ部レポート |
| アウトプット | Noteドラフト1本 / X投稿5本 / YouTubeショート台本3本 |
| 決済 | 社長が最終リライト（目安30分）し、公開承認する |

### 部署3: 営業部（SNS運用）

| 要件 | 内容 |
| --- | --- |
| 構成 | SNS担当エージェント |
| 制約（重要） | 最低3週間、社長が直接目視・投稿してからAIに引き継ぐ。即時自動化は禁止 |
| 引き継ぎ後動作 | 決済済み投稿をNote / Xに自動投稿（Codex in Chrome使用） |
| 凍結リスク対策 | 投稿前に社長確認ステップを必ず挟む |

### 部署4: カスタマーサポート部

| 要件 | 内容 |
| --- | --- |
| 構成 | CSエージェント |
| 対象 | Note / STORESの購入者 |
| アウトプット | 購入後自動メール送信 / 特典配布 / 質問一次返信 |
| エスカレーション条件 | クレーム・要望など重要案件のみ社長に通知 |

### 部署5: 社長秘書部

| 要件 | 内容 |
| --- | --- |
| 構成 | 秘書エージェント1 |
| 学習データ | 社長の過去Note記事・X投稿・著書 |
| 設置場所 | LINE Bot / Discord Bot |
| 機能 | 24時間Q&A対応・有料相談 / コンサルへの誘導 |

## 4. MDファイル設計要件

各エージェントに渡す定義ファイル（`.md`）には、以下の項目を含める。

```markdown
# [役職名]

## 役割
（例：制作部の編集長。記事ドラフトの管理と品質確認を担当する）

## 報告先
（例：社長）

## 管轄
（例：ライターA・ライターB・ライターC）

## 行動原則
- 成果物は必ず社長に決済を仰いでから次工程に進む
- 判断に迷う場合は上位に確認する

## 禁止事項
- 社長の決済なしに外部投稿を実行しない

## アウトプット形式
（例：Markdownレポート / JSON / 指定テンプレート）
```

必要な定義ファイルは以下の4種類とする。

- `社長.md`
- `部長.md`（部署ごと）
- `課長.md`
- `実行者.md`

## 5. 自動化スケジュール要件

| 時刻 | 処理 |
| --- | --- |
| 07:00 | リサーチ部起動 |
| 09:00 | 制作部起動（リサーチ部レポート決済済みの場合） |
| 随時 | カスタマーサポート部（購入イベントトリガー） |
| 随時 | 社長秘書部（LINE / Discord着信トリガー） |

実装方式は Codex Automation機能、またはClaude Code利用時のGitHub Actionsを候補とする。

## 6. 非機能要件

| 項目 | 要件 |
| --- | --- |
| SNS凍結リスク | 営業部は3週間の人間運用期間を必須とし、引き継ぎ後も投稿前確認ステップを維持 |
| メモリ蓄積 | 各エージェントに社長の修正ログを蓄積し、3ヶ月でパターン学習させる |
| コスト上限 | ChatGPT Pro月$20またはClaude Max内に収まるAPI使用量設計 |
| モバイル対応 | Codexモバイルから社長が外出先で指示を出せること |
| ブラウザ操作 | Codex in ChromeでNote / Xへの自動投稿を実現 |
| note規約対応 | 機械的な大量投稿・スパム投稿・外部ツールによる組織的な迷惑行為を禁止し、AI生成記事は社長の一次情報と目視リライトを経てから公開する |
| note収益化 | 無料記事で信頼構築後、有料記事・有料マガジン・メンバーシップ等を段階的に導入する。AI学習対価還元は副次収益として扱う |
| note / WordPress併用 | 初期はnoteを主戦場にし、反応が出た記事をWordPress保存版・SEO記事へ再編集して資産化する |
| バックアップ | note公開原稿と公式エクスポートデータを定期保存し、プラットフォーム依存リスクを下げる |
| AI風回避 | note記事はAIが完成させず、社長の体験談・失敗談・検証結果・感情の動きをAIが編集する形で制作する |
| 体験不足時の扱い | 実体験が不足する場合は、検証ログ・仮説記事・架空ケーススタディとして明示し、実体験のように偽装しない |
| ネタ選定 | noteランキング・人気タグ・売れ筋有料記事・SNS急上昇話題から、有料でも買う読者向けのテーマを選ぶ |
| シリーズ判断 | 反応がよい記事はシリーズ化し、閲覧・スキ・コメント・購入反応が落ちたテーマは撤退または保留にする |
| 画像活用 | 見出し画像・図解・チェックリスト画像・テンプレート画像を作成し、特に有料記事の商品価値を高める |
| 品質改善 | 読者設計、Before/After、一次情報、具体例、反対意見、図解、保存要素、投稿後改善を品質基準にする |

## 7. フェーズ設計

| フェーズ | 期間 | 内容 |
| --- | --- | --- |
| Phase 1 | Week 1-2 | リサーチ部 + 制作部のみ構築・動作確認 |
| Phase 2 | Week 3-5 | カスタマーサポート部追加 |
| Phase 3 | Week 6以降 | 営業部への引き継ぎ開始（3週間目視確認後） |
| Phase 4 | Month 3以降 | 社長秘書部 + メモリ蓄積による精度向上 |
| Phase 5 | Month 3以降 | note有料記事・有料マガジン・メンバーシップの小規模テスト |
| Phase 6 | Month 3以降 | note反応記事をWordPress保存版へ再編集 |
| Phase 7 | 継続 | 週次編集会議でシリーズ化・有料化・撤退を判断 |

## 留意点

- 営業部の即時自動化は設計上禁止とし、引き継ぎ条件（3週間目視運用）をシステムのチェックリストに組み込む。
- note投稿はAIによる量産ではなく、社長の一次情報・体験・具体例を含む記事制作に限定する。
- noteへの投稿前には `operations/note-compliance-checklist.md` を必ず確認する。
- note収益化は `operations/note-monetization-roadmap.md` に沿って、無料記事で信頼を作ってから段階的に行う。
- noteとWordPressの使い分けは `operations/note-vs-wordpress-strategy.md` に従う。
- note記事は月1回以上バックアップし、重要記事はローカル原稿も保存する。
- note記事の制作は `operations/note-article-writing-method.md` に従い、体験談・失敗談・具体例を必ず入れる。
- noteネタ選定は `operations/note-topic-selection-method.md` に従い、ランキング需要と有料購入意欲を見て決める。
- noteのシリーズ化・撤退判断は `operations/note-series-decision-rule.md` に従う。
- 「ボタン1つで月100万」は本システムの目標ではない。3ヶ月以上の継続稼働による複利的蓄積を前提とする。
- 初期構築は2部署から開始し、複雑性を段階的に上げる。
